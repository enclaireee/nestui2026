# NEST UI 2026 — Registration System + Admin Panel: Architecture Design (Phase 0)

## Stack (confirmed against repo)
- Next.js 16 (App Router, Turbopack), middleware file = `proxy.ts`.
- Supabase (Postgres + Auth + RLS), **new API-key system** (`sb_publishable_…` client key already in `.env.local`; server secret will be `sb_secret_…`).
- Existing user login already works (`lib/supabase/{client,server,proxy}.ts`) — registration hooks the logged-in session, not rebuilt.
- Deploy target: Vercel.

## Assumptions (flagged — correct me before Phase 2 if wrong)
1. **Competitions are the three already on the site**: `medhack`, `healthineer` (undergrad, **team size 3–5**), `healthynovation` (SMA, **team size 1–3**). Ranges taken from `app/branding/mainpage/section/competition.tsx`.
2. **Members have their own email.** The current UI omitted member email, but requirement §0.2 (“no duplicate emails between leader/members”) requires it. Added.
3. **Per-person field set**: `name`, `email`, `phone`, `student_id` (NIM for undergrad / NISN for SMA), `institution` (University / Sekolah), `major` (undergrad only, nullable), `confirmation_url` (the Google-Drive proof link the current form already collects per person).
4. **Team-level fields**: `team_name`, `competition`, `team_size`, `payment_proof_url`, `submission_url` (both from the existing “Submission” step), `status`, `code`, `submitted_at`.
5. **One team per user per competition** (`UNIQUE(user_id, competition)`) — prevents a single account spamming unlimited teams. A user may still enter different competitions.
6. **Statuses**: `pending` (default) / `verified` / `rejected`.
7. Reuse the existing WhatsApp/QR “Thank You” end screen.

## 0.1 Data model — decision: **one `registrations` table + one `team_members` table** (relational, not 3 tables)
**Why:** the three competitions are structurally identical (team → 1 leader + N members). A single `competition` enum column means **one** set of RLS policies, **one** insert function, **one** export view. Per-competition views/exports are a `WHERE competition = …` filter. Adding a 4th competition later = one enum value, zero schema change. Three physical tables would triple every policy, function, and view and make “export all” awkward. “Easy to see per competition” and “easy CSV per competition” are fully satisfied by filtering one column.

- `1 registration = 1 team = leader (stored on the registration row, 1:1) + N members (rows in team_members)`. `N = team_size − 1`.
- Enums: `competition_type ('medhack','healthineer','healthynovation')`, `registration_status ('pending','verified','rejected')`.
- Human-readable `code`: `NEST2026-<MDH|HTN|HNV>-0001`, generated by a BEFORE-INSERT trigger backed by one Postgres sequence per competition (atomic, concurrency-safe).
- Cross-column team-size rule enforced by a CHECK constraint **and** re-checked inside the insert function against the real member count.

See `BACKEND.md` for exact DDL.

## 0.2 Frontend flow — explicit 5-step state machine
Central state lives in `registration-client.tsx` via `useReducer` (one typed `RegistrationDraft` object). **Why useReducer over Zustand/URL params:** the wizard is a single mounted component tree, state is ephemeral until submit, and going Back must preserve everything — a reducer in the parent gives that for free with zero deps and no serialization concerns (payment/PII shouldn’t sit in the URL anyway).

1. **Competition select** — 3 cards (Medhack / Healthineer / Healthynovation).
2. **Team size select** — options driven by the chosen competition’s range (no hardcoded 2/3/4 branches; `N = size − 1` member forms).
3. **Leader form** — leader’s fields.
4. **Member forms (looped)** — paginated stepper “Member X of N”, loop count = `team_size − 1`. One reusable `<PersonForm>` used by both leader and members.
5. **Review & Submit** — read-only summary of everything, single submit.

- Back/forward preserves all state (reducer).
- Per-step client validation before “Next”: required fields, email format, phone format, **no duplicate emails** across leader+members, team-size/member-count integrity. Reuses `lib/registrations/validate.ts` (same functions run again server-side).
- Duplicate-submit prevention: submit button disabled while pending + server-side idempotency (`UNIQUE(user_id, competition)` → second submit rejected).

## 0.3 Backend / API
- Submission goes through a **Next.js Server Action** (`app/branding/registration/actions.ts`, `"use server"`) — never a client-side write. Server Actions get Next’s built-in same-origin/CSRF protection.
- Action flow: read logged-in user via `lib/supabase/server.ts` → **re-validate & sanitize every field server-side** (never trust client) → **rate-limit** (IP + user) → call Postgres RPC `submit_registration(...)` using the **secret key** (`lib/supabase/admin.ts`, server-only).
- `submit_registration` is a `SECURITY DEFINER` plpgsql function that: validates member count == `team_size − 1`, validates team-size range for the competition, rejects duplicate emails, inserts the registration + all members **in one transaction**, returns the generated `code`. Atomicity → no partial teams. `EXECUTE` granted to `service_role` only, so it cannot be called from the browser.
- All DB access is via the Supabase client / PostgREST / parameterized RPC args — **no raw SQL string-building anywhere** (submission or admin search). Classic injection is structurally impossible.
- Free-text (`team_name`, names) is HTML-stripped server-side before insert (defense-in-depth; React already escapes on render).
- Rate limiting: DB-backed `check_rate_limit(key, max, window_seconds)` (correct across serverless instances, unlike an in-memory Map). Used by submission and admin login.

## 0.4 Security
- **RLS enabled on every table**, deny-by-default. `anon`+`authenticated` (the publishable key’s roles) get **no** INSERT/UPDATE/DELETE on registration tables; a narrow SELECT-own policy lets a user view only their own team (`user_id = auth.uid()`). All writes happen with the secret key (bypasses RLS) inside the server action, or via the `SECURITY DEFINER` RPC.
- Secret key (`sb_secret_…`) used **server-side only**, never `NEXT_PUBLIC_`, never shipped to the browser. Guarded by an `import "server-only"` in `lib/supabase/admin.ts`.
- `auth.uid()` ties each registration to its owner; `UNIQUE(user_id, competition)` caps one team per competition per user.
- Input sanitization: strip/escape HTML on free-text → no stored XSS in the admin panel.
- CSV export escapes formula-injection (`= + - @` prefixed with `'`).
- CSRF: Server Actions validate origin by default; admin login is also a Server Action.
- **Admin auth (`nest2026` / `ADMIN123`)** — separate from Supabase Auth:
  - Credentials in **env vars** (`ADMIN_USERNAME`, `ADMIN_PASSWORD`), not committed literals.
  - After login, a **signed httpOnly, Secure, SameSite=Lax cookie** — an HMAC-SHA256 (Web Crypto) token of `{exp}` signed with `ADMIN_SESSION_SECRET`. **Not** a spoofable `isAdmin=true` flag. Constant-time signature compare.
  - All `/admin/*` pages + admin API routes verify the cookie server-side (`requireAdmin()` guard), plus a `proxy.ts` redirect for UX.
  - Admin login **rate-limited** (DB) to blunt brute force against the known credential.
  - **Accepted tradeoff, explicitly noted:** this is a single shared credential with **no per-user audit trail** — chosen per requirement, not recommended for multi-admin use.
  - Admin data access uses the **secret key** server-side (must bypass RLS to see all teams).
- File uploads: the flow collects **Google-Drive links** (text), not direct file uploads, so no storage bucket is needed. If real uploads are added later: private bucket + server-side MIME/size validation (noted, out of current scope).

## 0.5 Admin panel (`/admin`)
- `app/admin/login` — username/password Server Action → verify env creds (constant-time) → set signed cookie. Rate-limited.
- `app/admin/(protected)/` route group guarded by `layout.tsx` (`requireAdmin()` → redirect to `/admin/login`):
  - `page.tsx` — dashboard: per-competition tabs/filter, paginated table (team name, competition, member count, submitted date, status), search by team name / code / leader email (parameterized).
  - `registrations/[id]/page.tsx` — full leader + all members detail.
- `app/admin/export/route.ts` — CSV export, `requireAdmin()` inside the handler. **Flattened, one row per team** (`Code,Team,Competition,Status,SubmittedAt,Leader_Name,Leader_Email,…,Member1_Name,Member1_Email,…`). **Why flattened:** one team = one row is the “easy to see / open in Excel” shape stakeholders expect; a joined multi-row export splits a team across rows and complicates counting. Column count = leader + `max(team_size)−1` members. `?competition=` filter + “export all”. Proper `Content-Disposition: attachment; filename=…csv`. Formula-injection-escaped. Blocked without admin session.
- Logout Server Action clears the cookie.
- Pagination: `.range()` server-side (never fetch all rows); indexes on `competition`, `submitted_at`, `status` back the sort/filter.

## 0.6 Performance / UX
- Validation on blur/submit, not on every keystroke (no excessive re-renders).
- Admin table paginated via `.range()` + indexed columns.
- Submission shows explicit loading/success/error (no optimistic UI for a competition entry).
- Indexes: `competition`, `submitted_at`, `status`, `user_id`, `team_members(registration_id)`, unique `(competition, lower(leader_email))`.
